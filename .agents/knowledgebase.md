## Project shape

Flutter desktop IDE (Windows / macOS / Linux), Dart-only `lib/`. Code intelligence is provided by GitNexus — see `AGENTS.md` / `CLAUDE.md` at repo root for the always-do/never-do rules. The index is sometimes stale; verify with `gitnexus://repo/lumen/context` and run `npx gitnexus analyze` if needed.

Top-level layout:
- `lib/providers/` — `AppState` (workspace, settings, view mode) + `ChatController` (messages, models, tools).
- `lib/services/` — service-per-concern: `ollama_service.dart`, `gemini_service.dart`, `anthropic_service.dart`, `github_models_service.dart`, `chat_persistence_service.dart`, `timeline_service.dart`, `syncthing_service.dart`, `gitnexus_service.dart`, etc.
- `lib/widgets/` — UI. `widgets/ai_chat/ai_chat.dart` is the chat panel (model picker lives here as `_ModelPicker`).
- `lib/l10n/strings.dart` — single-file string catalog. **All hardcoded text must go through `S.<name>`.**
- `lib/theme/` — `DuckColors`, `DuckTheme`. Use `DuckColors.*` and `DuckTheme.radius*` everywhere.

## LLM provider model — important contracts

`ChatController` is the single owner of:
- `_availableModels: List<String>` — `provider:rawModel` ids fetched from each enabled provider
- `_enabledModels: Set<String>` — user-curated subset shown in the picker
- `_selectedModel: String` — the currently routed model (full `provider:rawModel`)

Three derived getters drive the UI:
- `availableModels` — full fetched list
- `enabledModels` — user-enabled subset
- `pickerModels` — `availableModels ∩ enabledModels`, the dropdown contents

**Refresh entry points** (in order from most → least invasive):
1. `chat.init()` — boot
2. `chat.reloadModels({enabledProviders})` — called from:
   - `AppState.updateProviderSettings` (Settings → Save / LLM provider dialog → Save)
   - `AppState.resetGithubUnavailableModels`
   - GitHub auto-discovery callback (`onUnavailableModelDiscovered`)
   - `_OllamaSetupDialog._runCheck` when daemon transitions to reachable *(added 2026-05)*
   - Model picker dropdown → "Refresh models" *(added 2026-05)*

If you add a new place where the model surface changes (new provider, key change, daemon state flip), wire `chat.reloadModels()` into it. Without it the picker is stale until next save.

## GitHub Copilot provider

Copilot is a separate provider namespace from GitHub Models:
- `github:<publisher/model>` = GitHub Models REST inference (`https://models.github.ai`).
- `copilot:<model>` = GitHub Copilot via the local Node bridge and `@github/copilot-sdk`.

The Copilot bridge lives in `assets/bin/copilot/` and is materialised by `lib/services/copilot_provisioner.dart` into `<appSupport>/bin/copilot/`. The provisioner copies `bridge.js` + `package.json`, runs `npm install --omit=dev --no-audit --no-fund` only when the asset sha changes or `node_modules/` is missing, then returns the real `bridge.js` path. Node must be on PATH; this is the same broad prerequisite as GitNexus.

Runtime flow:
- `CopilotService` owns one local `node bridge.js` child process and talks line-delimited JSON over stdio. No TCP server is opened.
- Auth precedence: non-empty `copilot.apiKey` -> SDK `githubToken`; otherwise `copilot.useLoggedInUser == true` -> SDK `useLoggedInUser:true`.
- First-time setup: Settings -> General -> GitHub Copilot has a "Sign in to Copilot" button. It provisions the local bridge, opens the bundled `node_modules/.bin/copilot` CLI in a real terminal, and the user runs `/login` there. This avoids requiring a global `npm install -g @github/copilot`; Lumen's app-support install is enough.
- Model discovery must use the SDK's real `client.listModels()` result. Do not silently fall back to a hardcoded model list when authenticated; that masks org/policy/auth failures and makes Settings report fake success.
- `ChatController` routes `copilot:` models through `CopilotService.generateChatStream`, title summaries through `summarizeTitle`, and marks Copilot as native-tool capable.
- Native tool calls: the bridge registers SDK `defineTool` handlers for Lumen's JSON schemas, emits a `tool_call` event to Dart, and waits. `CopilotService` converts that to `NativeToolUseMarker`, asks the bridge to cancel the current SDK turn, and lets Lumen's existing native-tool loop execute the tool + include the tool result in the next prompt iteration. This keeps approvals, timeline logging, and single-tool-per-iteration behaviour inside Lumen.
- Provider display helpers must stay in sync: `widgets/ai_chat/ai_chat.dart::_providers/_prettyProvider` and `widgets/ai_chat/model_management_panel.dart::_providers/_prettyProvider` both include `copilot` after `github`.

## Ollama specifics — the thing that bit us

`OllamaService.getModels()` previously returned `['llama3','mistral','phi3']` as a fallback when `/api/tags` failed. This produced ghost entries in the picker on PCs without Ollama running and was the root cause of "models didn't show up after setting endpoint" reports.

**Current behavior (2026-05 onward):**
- `getModels()` returns `[]` on any failure (unreachable / non-200 / malformed JSON).
- The picker's `S.chatNoModels` empty state handles the "nothing available" case.
- `isReachable()` is the canonical "is the daemon up?" probe; positive `isReachable` + empty `getModels` = "running but no models pulled".
- Cloud Ollama models are heuristically detected by `OllamaService.isCloudModel(rawModel)` — suffix `-cloud` or `:cloud`. Cloud models auto-pin `num_ctx`; do NOT send a `num_ctx` option for them.

**Dual-mode Ollama: local daemon + Cloud API key, separate namespaces (2026-05 onward).** `OllamaService` now carries an optional `apiKey` field (persisted as `ollama.apiKey`, plumbed through `AppState._ollamaApiKey`). When non-empty, the service ALSO talks to `https://ollama.com` (`OllamaService.cloudBaseUrl`) with `Authorization: Bearer <key>`. The two paths surface as **two distinct provider namespaces** in the picker so the Model Management panel renders them as separate tabs:
  - `ollama:<name>` ← local daemon (`getModels()` against `$baseUrl/api/tags`). Includes user `ollama pull`'d models, including any `*-cloud` ones added via `ollama signin` SSO and proxied through the local daemon.
  - `ollama-cloud:<name>` ← Ollama Cloud API key path (`getCloudModels()` against `https://ollama.com/api/tags` with Bearer auth). Names are returned bare (no `-cloud` suffix); the suffix is stripped via `stripCloudSuffix` because the namespace prefix already carries the route information.
  - **Dedupe rule** in `ChatController._fetchModels`: when `hasCloudApiKey && cloudModels.isNotEmpty`, drop any `*-cloud`/`*:cloud` entries from the local daemon list — they'd be functionally equivalent to the `ollama-cloud:` entries and showing the same model in two tabs is just confusing. Pure local models stay. Without a cloud key, the legacy `ollama signin` SSO path through the local daemon keeps working untouched.
  - **Routing** is centralised in `OllamaService._resolveChatRoute(rawModel, {forceCloud})`. `forceCloud=true` → `cloudBaseUrl` + auth + `stripCloudSuffix` (used for everything in the `ollama-cloud:` namespace). Otherwise: heuristic — `hasCloudApiKey && isCloudModel(rawModel)` also routes to cloud (legacy back-compat for users who picked an `ollama:foo-cloud` model from the local namespace and have a key configured). Otherwise local. `generateChat`, `generateChatStream`, and `summarizeTitle` all take `forceCloud` and pass it through. `ChatController._splitModel('ollama-cloud:gpt-oss:120b')` → `('ollama-cloud', 'gpt-oss:120b')` because we split on the FIRST `:` and the pretty name has no inner colon issue (the raw name's own colons survive in the second tuple element).
  - **Provider-name mapping**: both `ollama:` and `ollama-cloud:` map to the single `'Ollama'` enabled-provider toggle (the `_ => 'Ollama'` default in the routing switch). The split is purely for picker grouping and route selection — there's no separate "Ollama Cloud" toggle in Settings → General.
  - **Display labels**: `S.providerOllama = 'Ollama'`, `S.providerOllamaCloud = 'Ollama Cloud'`. Both `widgets/ai_chat/ai_chat.dart::_prettyProvider/_providers` and `widgets/ai_chat/model_management_panel.dart::_prettyProvider/_providers` register `ollama-cloud` adjacent to `ollama` in the sort order. **If you ever add another Ollama-derived namespace (e.g. a custom remote daemon), update both helpers in lockstep — they're independently maintained today.**
  - `ChatController.isReachable` returns true for "Ollama" if EITHER `ollama.hasCloudApiKey` OR `ollama.isReachable()`. The first-run heuristic in `welcome_screen.dart::_isLumenFirstRun` and `menu_bar.dart::_isLumenFirstRun` similarly counts a non-empty `state.ollamaApiKey` as "user has configured a provider".
  - UI: `lib/widgets/llm_providers_setup_dialog.dart` (wizard) and `lib/widgets/settings_view.dart` (Settings → General → Ollama) both expose a Cloud API key field next to the Local endpoint field. `updateProviderSettings` gained a required `ollamaApiKey` parameter — every call site (`settings_view.dart`, `llm_providers_setup_dialog.dart`, `settings_dialog.dart` legacy) passes it.

**Onboarding wizard sequence** (`welcome_screen.dart::_runNewProjectWizard` and `menu_bar.dart::_runNewProjectWizardFromMenu` — keep in sync):
1. `showOllamaSetupDialog` (first run only)
2. `showOllamaCloudKeyPromptDialog` (first run only AND `state.ollamaApiKey.isEmpty`) — narrow single-field prompt; seeds a cloud key specifically so the next step (skill generator) can route through a frontier cloud model. Skip is non-destructive.
3. `showLlmProvidersSetupDialog` (first run only)
4. `showSkillGeneratorDialog`
5. `showGitNexusOnboardingDialog`
6. Syncthing prompt

`_isLumenFirstRun` heuristic = no API keys saved AND Ollama not reachable.

**Skill generation model selection** (`lib/services/skill_model_picker.dart::pickSkillModel`). Skill generation is a one-shot JSON-shaped task; whatever the user has set as `chat.selectedModel` for streaming chat is rarely the right tool. The picker:
- If `state.ollamaApiKey` non-empty → query `OllamaService.getCloudModels()` and pick the first match in `['qwen3-coder:480b', 'gpt-oss:120b', 'deepseek-v3.1:671b']` as `'ollama-cloud:<name>'`. If none of those are in the user's catalog but any cloud model is, picks the first cloud model rather than dropping to a local one (the user pasted a cloud key — respect intent). On any failure (catalog fetch timeout, etc.) falls back to `selectedModel`.
- If no Ollama key → returns `selectedModel` untouched (so a Gemini-configured user routes through their selected Gemini model, etc.).
- Both wizard skill dialog (`showSkillGeneratorDialog::_startGenerate`) AND manual skill dialog (`showManualSkillDialog::_create`) call `pickSkillModel` before constructing `SkillGenerator`. Update both call sites in lockstep if the picker contract changes.

**Skill generator prompt** (`lib/services/skill_generator.dart`) was significantly rewritten in 2026-05 to lift output quality from "generic bullet-point platitudes" to "senior-engineer-onboarding-cheatsheet". Key invariants in the system prompt + `_schemaAndRules` block:
- **Quality bar via contrastive examples** — embedded GOOD vs BAD skill in the prompt body so the model anchors on specifics.
- **Trigger discipline** — explicit ban on `'always'` / `'when coding'` / `'when relevant'` triggers (at most one `'always'` skill per response, and only for project-wide invariants).
- **Project-grounding requirement** — every skill must reference at least two concrete things from the project context (file paths, manifest entries, idiomatic framework usages).
- **Anti-pattern checklist** — explicit rejection list (don't paraphrase framework docs, don't explain builtin tools, don't emit shell-script tools that are really LLM problems, etc.).
- **Quantity guidance** — output 2-5 artifacts not 3-7. Self-check pass before emitting JSON. The validator unchanged; it accepts any number of valid artifacts so a single great skill is preferred over five mediocre ones.
- The schema (tool/skill JSON shape, snake_case ids, 8000-char body cap, regex-required-for-tool-pattern) is unchanged so `_parseValidateAndWrite` and `WorkspaceSkillsService.compileForPrompt` consume the same shape they always have.

## Workspace conventions

- **i18n always.** New text → add a `static const String` to `lib/l10n/strings.dart`, reference as `S.<name>`. No hardcoded literals in widgets.
- **Modular files.** No 20k-line monoliths; split when a file gets unwieldy.
- **Global + page-specific styling.** Use `DuckColors.*` / `DuckTheme.*` first; only override per-element when needed.
- **No webservers without explicit user consent.** Most surfaces are auth-gated.
- **OS = Windows.** Default shell is PowerShell. Quote paths with spaces; prefer `Test-Path` / `New-Item` over `if not exist mkdir`.
- **Conda-activate for Python.** When running Python tooling, activate the project env first.

- **2026-05 — Ollama tool-calling protocol fix (post-overhaul correction).** The native-tools rollout's first cut of `OllamaService.generateChatStream` was broken in three ways that explain why glm-5.1, gemma4:31b and similar models cycled through auto-continues without ever firing a tool. **Re-read the docs at https://docs.ollama.com/capabilities/tool-calling whenever you touch this code path.** What was wrong:
  - **Tool_calls only checked on `done:true`.** The docs explicitly say "When streaming, gather every chunk of `thinking`, `content`, and `tool_calls`". Lumen's first cut copied OpenAI's pattern and only inspected the final frame. Result: when an Ollama model emitted a tool_call mid-stream, we threw it away, and the `done` frame's tool_calls field was empty (because the model had already flushed). The user saw "stopped instantly" with thinking + prose + no tool. Fix: `seenToolCalls` flag in the per-chunk loop emits a `NativeToolUseMarker` for the first tool_call we see and `return`s (early stream close); the done-frame check stays as a defensive fallback gated on `!seenToolCalls`.
  - **Wrong tool reply field.** Docs use `{role: 'tool', tool_name: 'get_temperature', content: '22°C'}` — the `tool_name` field carries the function NAME, not a synthesized call id. Lumen had copied OpenAI's `tool_call_id` shape with our random hex id, so Ollama couldn't link replies back to calls. Fixed by stamping both `tool_use_id` AND `tool_name` on every tool reply in `apiMessages` (controller side), and Ollama's translator picks `tool_name`. Anthropic / OpenAI translators continue using `tool_use_id`.
  - **`think` is NOT something Lumen needs to set explicitly.** First cut sent `think: true` whenever `/api/show` reported `thinking` capability. Wrong on two counts per https://docs.ollama.com/capabilities/thinking: (a) "Thinking is enabled by default in the CLI and API for supported models" — sending `true` is redundant noise; (b) GPT-OSS *ignores* boolean `think` and expects string levels (`"low"` / `"medium"` / `"high"`). Final shape: omit the field entirely. **If we ever want to tune reasoning intensity** for the GPT-OSS family specifically, add a `model.startsWith('gpt-oss')` switch that maps `ReasoningEffort` → `"low|medium|high"` and only sets `think` for that family. For every other thinking-capable model the default is correct.
  - **Spurious outer `id` on tool_calls.** The docs' assistant-turn shape is `tool_calls: [{type: 'function', function: {index, name, arguments}}]` — no outer `id`, just `index` inside `function`. Lumen had emitted `{id: ..., type: 'function', function: {name, arguments}}`. Fixed in `_translateOneOllamaMessage`. Probably tolerated by Ollama anyway, but matching the docs verbatim is the safer place.
  - **Mid-stream tool_call emission must close any open thinking phase first.** The controller's marker state machine routes thinking tokens to `thinkBuf`. If we emit a `NativeToolUseMarker` while `inThinking` is true, the live message stays stuck in "thinking…" mode. Fix: emit `thinkEndMarker` (and the inline-`<think>` close) before yielding the tool_use marker.
  - **Why this matters more than expected.** After the Anthropic/Gemini/OpenAI native paths landed in mid-2026-05, the user reported gemma4:31b and deepseek-v4-pro "stopping instantly" — and we initially diagnosed it as model-side laziness needing a stronger detector. The detector expansion (`understand`, `review`, `explore`, `start`, etc. added to `_actionVerbs`) is good and stays, but it was treating a SYMPTOM. The actual root cause was that we simply weren't reading the tool_calls Ollama was streaming us. **Lesson for future agents touching provider services: read the provider's tool-calling docs end-to-end — including the streaming section — before writing a stream parser. Each provider's stream model differs (Anthropic per-block content_block_start/stop, OpenAI partial_json deltas keyed by index, Gemini inline function_call parts, Ollama complete tool_calls per chunk that you `.extend()`).**

- **2026-05 — Native tool calling, model tiering, Ollama Cloud 182s pre-cut, detector consolidation (the "chat dies" overhaul).** Largest single change to the chat stack since project inception. Driven by user reports of Claude turns silently dying mid-stream and weak Ollama models looping on EDIT_FILE near-misses. Replaced the universal text-grammar `<<<TOOL>>>` protocol with provider-native tool calling on **every** capable provider while keeping the legacy path alive for genuinely incapable models.
  - **Native tool dispatch end-to-end.** Each provider service (`anthropic_service.dart`, `github_models_service.dart`, `gemini_service.dart`, `ollama_service.dart`) now accepts a `nativeToolIds: Set<String>?` and translates the `tools[]` array into its wire shape (Anthropic `input_schema`, OpenAI `function`, Gemini `function_declarations`, Ollama OpenAI-compatible). Stream parsers were extended to assemble streamed `tool_use` / `tool_calls` deltas (Anthropic SSE per-block, OpenAI partial_json deltas, Gemini inline `function_call` parts, Ollama final-frame `tool_calls`) and emit a synthetic `<!-- LUMEN_NATIVE_TOOL_USE:<base64-json> -->` marker that the controller's chunk loop catches.
  - **Synthetic Match adapter** (`lib/services/tools/tool_match_adapter.dart` + `tool_schemas.dart`). - **2026-05 — SSH integration v1.1 (typing fix, dropdown picker, right-slot landmine).** Three bugs the v1 ship hit in real use, all fixed in one pass. Future agents working in this area: read this BEFORE you touch the right-slot layout, the picker, or anything xterm-related, because all three traps have non-obvious failure modes.
  - **`xterm` keyboard input on desktop ⇒ `hardwareKeyboardOnly: true` is mandatory.** xterm 4.x has two input paths: the default `CustomTextEdit` (IME-backed via Flutter's `TextInputConnection`) and `CustomKeyboardListener` (raw `keyEvent.character`). The default keytab has **no entries for plain letters/digits/punctuation** (`lib/src/core/input/keytab/keytab_default.dart` only defines control / arrow / function / mod-combo bindings) so `terminal.keyInput(TerminalKey.keyA, ctrl:false)` returns `false`. With the default path, a printable `'a'` is supposed to come back through the IME's `updateEditingValue` → `onInsert` → `terminal.textInput` → `Terminal.onOutput`. On Windows desktop the `TextInputConnection` (a) only opens after an explicit `requestKeyboard()` (xterm calls it on tap-down) AND (b) races with the chat panel's `TextField` and the editor's `re_editor` connections. Symptom: Ctrl+C / Ctrl+L / arrows / Enter all worked (resolved synchronously by `CtrlInputHandler` against `HardwareKeyboard.instance`), but plain letters were silently swallowed. Fix lives in `lib/widgets/ssh/remote_pane.dart::_SessionTerminalView` — `hardwareKeyboardOnly: true` + `autofocus: true`. The hardware path's wrapper (`CustomKeyboardListener._onKeyEvent`) reads `keyEvent.character` directly and pumps it through `onInsert` without any IME dependency. **Trade-off:** loses IME composition, so CJK / dead-key composition won't work in the SSH terminal — explicit choice for a Windows-first IDE; revisit by giving the terminal its own `FocusNode` and post-frame `requestKeyboard()` if CJK input matters later. - **2026-05 — SSH v1.4.2: shell helpers auto-installed via SFTP-uploaded self-deleting script.** The manual "install shell helpers" icon (terminal_outlined chrome button in `RemotePane`) and its modal (`lib/widgets/ssh/ssh_shell_helpers_dialog.dart`) were retired. `lumen-edit`, `lumen-grab`, and the OSC 7 cwd reporter now auto-install into every fresh SSH session via the **SFTP-upload-then-source** pattern: `SshController._runConnect` schedules a 250ms-delayed task that SFTP-writes the helper body to `/tmp/.lumen-helpers-<rand>.sh`, then types `. <path>` into the interactive shell. The script's last statement is `rm -f -- "$__lumen_self"` so it self-cleans after sourcing. Net visible terminal traffic is one short `.`-source line + a small duck banner ("Lumen connected · helpers: …") — banner / motd / first PS1 are preserved.
  - **Why upload-and-source instead of typing the body inline.** dartssh2's shell channel runs in canonical (cooked) mode → the remote shell echoes every byte we write through stdout. Typing a 700-char one-liner dumps a wrapped-across-5-lines blob into the user's freshly opened terminal. Uploading the body to a file means the user only sees ONE short input line; the function definitions stay invisible because they live in the file we wrote, not in the input stream. The duck banner that the script prints on its way out is intentional "hello, helpers loaded" feedback rather than syntax noise. Rejected alternatives the user explicitly killed: (a) `clear` after the inline blob — feels like "sneaking" the inject and wipes the OS banner, (b) the verbose multi-line `allShellHelpers()` paste — even more echo than the one-liner.
  - **Where the script body lives.** `autoInstallShellHelpersScript()` in `lib/services/ssh/ssh_terminal_pre_processor.dart`. Dart raw triple-quote string (`r'''…'''`) so the backslashes (`\033`, `\a`) and shell vars (`$1`, `$PWD`, `${BASH_SOURCE:-$0}`) survive into the wire verbatim — no Dart-side `\$` doubling. **If you change this script:** keep `rm -f -- "$__lumen_self" 2>/dev/null` as the LAST executable statement (bash buffers/parses the body before executing, so file deletion mid-script is safe only because no later statements need to be re-read from disk). Re-test self-delete by running `ls /tmp/.lumen-helpers-*` after a connect — should return nothing.
  - **Self-delete cross-shell trick.** `__lumen_self="${BASH_SOURCE:-$0}"` works in both bash (uses `BASH_SOURCE`) and zsh (`BASH_SOURCE` is unset → falls back to `$0`, which zsh sets to the sourced filename by default). Confirmed against `bash 5.x` on Ubuntu and `zsh 5.x` on Arch / macOS.
  - **Per-session unique path.** `/tmp/.lumen-helpers-<microsSinceEpoch base36>.sh`. Two concurrent Lumen sessions to the same host (or a reconnect arriving before /tmp's reaper runs) can't collide on the helper path. The base-36 microsecondsSinceEpoch is 8–9 chars and unique per Dart process tick.
  - **Failure modes — all silent, all bail with `debugPrint`.** `conn.sftp()` throws → server has no SFTP subsystem (rsync-only / locked-down hosts). `sftp.open` throws → `/tmp` is readonly or quota-exceeded. Shell channel closes between SFTP write and `pasteIntoSession` → caught inside `pasteIntoSession` itself. **In all three cases the user gets a working SSH shell sans helpers**, which is strictly better than dumping installer noise into their terminal. There's no fallback to inline-paste — if SFTP is unavailable, the user can run `lumen-edit` manually from a copy/paste they keep elsewhere.
  - **Shell compatibility inside the script body.** The functional body is guarded by `[ -n "$BASH_VERSION$ZSH_VERSION" ]`. fish / dash / csh / busybox don't enter the body but still run the banner + self-delete portion (banner uses POSIX `cat <<'…'`, self-delete uses POSIX `rm`). For exotic shells where `cat`/`rm` aren't on PATH the script just prints whatever errors the shell emits and the invocation line lands in the user's history — acceptable for the v1 audience (Linux/macOS targets).
  - **Idempotency on reconnect.** The bash branch wraps the `PROMPT_COMMAND` prepend in `case "$PROMPT_COMMAND" in *_lumen_osc7*) ;; *) PROMPT_COMMAND="_lumen_osc7${PROMPT_COMMAND:+;$PROMPT_COMMAND}" ;; esac`. The zsh branch wraps `precmd_functions+=(_lumen_osc7)` in a parallel guard. Function definitions (`lumen-edit()`, `lumen-grab()`) are naturally idempotent — re-defining is a noop. The `${PROMPT_COMMAND:+;$PROMPT_COMMAND}` parameter expansion only emits the `;<old>` chain when `$PROMPT_COMMAND` is non-empty, so a fresh shell with no prior PROMPT_COMMAND ends up with `_lumen_osc7` (no trailing semicolon) instead of `_lumen_osc7;` (which some shells warn about).
